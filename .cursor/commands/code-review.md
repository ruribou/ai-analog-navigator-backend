# code-review

## 目的
指定された差分やファイル群に対して、AGENTSガイドおよびプロジェクトルールに沿ったコードレビューを実施し、問題点を優先度順に指摘する。

## 引数
`/code-review <diffや対象ファイルのパス、説明Markdown>`

## 実行フロー
1. 引数からレビュー対象（ブランチ名、変更ファイルリスト、補足Markdownなど）を特定し、必要に応じてユーザーへ追加情報を確認する。  
2. 関連するPlan/Taskドキュメントがある場合は内容を引用し、要求仕様・成功条件を整理する。  
3. 変更ファイルを読み込み、以下の観点で問題を洗い出す:
   - バグや例外パス、入力バリデーション漏れ
   - 仕様逸脱やPlan/Taskとの不一致
   - パフォーマンス/セキュリティ/可用性リスク
   - テスト・検証不足、ドキュメント未更新
4. レビュー結果は以下のテンプレートに沿って整理し、カテゴリごとに箇条書きで記述する:
   ```
   # コードレビュー結果

   ## 変更ファイル一覧
   - path/to/file1
   - ...

   ## 🔴 Critical（要修正）
   - ファイル: 問題と推奨修正

   ## 🟡 Warning（推奨修正）
   - ...

   ## 💡 Info（改善提案）
   - ...

   ## 🟢 Good
   - 良い点
   ```
5. 必要ならオープン質問や追加確認事項をテンプレ内のカテゴリに追記し、最後に残リスクやテスト不足をWarning/Infoへ反映する。  
6. すべて日本語で、簡潔かつ actionable なフィードバックとして報告する。

## 注意事項
- レビューは建設的に行い、根拠を明記する。  
- 指摘がない場合も、確認済み範囲と残る懸念を明示する。  
- 大量のファイルがある場合は優先度の高いファイルから着手し、必要に応じて範囲の絞り込みを提案する。***
